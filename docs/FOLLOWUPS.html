Followups

Backlog spillover discovered during implementation. Keep items PR-sized when possible.

Critical convention. Every followup must carry a data-priority attribute. Three buckets:
  • blocks-release: cannot ship v1 without this.
  • nice-to-have: improves the product but does not block.
  • polish: post-release cleanup.

How to add a followup

<section data-f="F-NNN" data-priority="blocks-release|nice-to-have|polish">
  <h3>F-NNN: Short title</h3>
  <dl>
    <dt>Context</dt><dd>one or two sentences on why this came up.</dd>
    <dt>Blocker (if any)</dt><dd>the condition that prevents working on this now.</dd>
    <dt>Unblock condition</dt><dd>what has to be true to start.</dd>
    <dt>PR / Dot reference</dt><dd>#N or dots-N (when picked up)</dd>
  </dl>
</section>

Keep F-NNN IDs monotonically increasing. When a followup ships, leave the entry in place and append a <dt>Resolved</dt><dd>PR #N</dd> entry. Never delete.

F-006: Throw spin/angle meter (REQ-034) and release-to-launch (REQ-036)

Context
Step 2 of the control scheme: after locking the line, a sweeping meter sets spin/angle (centre is straight, sides curve). On release this must feed the launch: the lineup lateral position plus the chosen angle/spin combine into the ball's launch velocity and angular velocity (REQ-036), so the ball curves. Today the ball launches straight down-lane from the aligned x; this slice makes aim and spin actually steer the shot. Meter sweep speed and sweet-spot width start forgiving and tighten via the playtest gate (Q-010), and render as mechanical gauges (REQ-038).
Blocker
none; the lineup (F-005 / REQ-033) and the rolling ball (REQ-029) exist.

F-008: The playable game loop (throw, detect, score, reset, frames, end)

Context
Wire the existing systems into an actual game: after the throw, wait for the rack to settle, run detection (src/detection.ts) to count this ball's pinfall, feed it into the scoring engine (src/scoring.ts scoreGame), run the reset cycle (src/reset.ts) to reel fallen pins and re-rack between frames, advance through three balls per frame and ten frames, and show the end-of-game summary (REQ-001, REQ-002, REQ-008, REQ-009, REQ-010, REQ-011). The pieces all exist and are unit tested in isolation; this is the orchestration slice that makes VibePins a game. A visible scoreboard (REQ-012, REQ-042) can land here or as a paired UI slice.
Blocker

Unblock condition
scoring, detection, and reset are merged (they are); wire them together.
Progress
The pure orchestration spine landed as src/game.ts (the Game class): it consumes per-ball pinfall, tracks frame and ball, leaves standing pins between balls and re-racks at frame end, runs the tenth-frame bonus-ball logic, and ends the game with a summary (REQ-009, REQ-010, REQ-011 now partial). What remains for F-008: wire it into src/main.ts so detection feeds recordBall, the returned between-balls / rerack reset drives ResetCycle / PinSet, and the game-over summary renders on screen. The visible scoreboard (REQ-012, REQ-042) can land with the wiring or as a paired UI slice.

F-004: Ball containment (gutters and a back pit)

Context
The ball slice launches the ball straight down-lane on the bed (x=0), so it stays on the lane and reaches the pins. But there are no gutters (REQ-031) on the sides and no pit/backstop behind the pin deck, so a ball with any lateral component would roll off the side and a ball that clears the rack rolls off the back of the deck and falls into the void (the ball-roll smoke confirms this). Both are scene/physics additions.
Blocker
none to start; it has no value until the ball can travel sideways.
Unblock condition
This becomes a hard prerequisite for the three-step control scheme (REQ-033, REQ-034 lateral aim and spin): gutters must exist before the ball can be aimed off-centre, and a back pit/backstop should exist so a thrown ball comes to rest rather than falling forever. Implement gutters (REQ-031) and a simple back pit before or alongside the lateral-aim controls slice.
PR / Dot reference
(not started)
