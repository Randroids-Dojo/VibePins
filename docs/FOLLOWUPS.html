Followups

Backlog spillover discovered during implementation. Keep items PR-sized when possible.

Critical convention. Every followup must carry a data-priority attribute. Three buckets:
  • blocks-release: cannot ship v1 without this.
  • nice-to-have: improves the product but does not block.
  • polish: post-release cleanup.

How to add a followup

<section data-f="F-NNN" data-priority="blocks-release|nice-to-have|polish">
  <h3>F-NNN: Short title</h3>
  <dl>
    <dt>Context</dt><dd>one or two sentences on why this came up.</dd>
    <dt>Blocker (if any)</dt><dd>the condition that prevents working on this now.</dd>
    <dt>Unblock condition</dt><dd>what has to be true to start.</dd>
    <dt>PR / Dot reference</dt><dd>#N or dots-N (when picked up)</dd>
  </dl>
</section>

Keep F-NNN IDs monotonically increasing. When a followup ships, leave the entry in place and append a <dt>Resolved</dt><dd>PR #N</dd> entry. Never delete.

F-008: The playable game loop (throw, detect, score, reset, frames, end)

Context
Wire the existing systems into an actual game: after the throw, wait for the rack to settle, run detection (src/detection.ts) to count this ball's pinfall, feed it into the scoring engine (src/scoring.ts scoreGame), run the reset cycle (src/reset.ts) to reel fallen pins and re-rack between frames, advance through three balls per frame and ten frames, and show the end-of-game summary (REQ-001, REQ-002, REQ-008, REQ-009, REQ-010, REQ-011). The pieces all exist and are unit tested in isolation; this is the orchestration slice that makes VibePins a game. A visible scoreboard (REQ-012, REQ-042) can land here or as a paired UI slice.
Blocker
best after the throw is player-driven (F-006/F-007) so real pinfall flows in, though it can be prototyped with the current fixed launch.
Unblock condition
scoring, detection, and reset are merged (they are); wire them together.
Progress
The pure orchestration spine landed as src/game.ts (the Game class): it consumes per-ball pinfall, tracks frame and ball, leaves standing pins between balls and re-racks at frame end, runs the tenth-frame bonus-ball logic, and ends the game with a summary (REQ-009, REQ-010, REQ-011 now partial). What remains for F-008: wire it into src/main.ts so detection feeds recordBall, the returned between-balls / rerack reset drives ResetCycle / PinSet, and the game-over summary renders on screen. The visible scoreboard (REQ-012, REQ-042) can land with the wiring or as a paired UI slice.
PR / Dot reference
orchestrator: PR #TBD; physics/UI wiring: (this slice)
Resolved
The physics/UI wiring landed in src/main.ts as an explicit shot state machine (aiming, watching, settling, resetting, over). After the three-step throw the new ShotWatcher (src/shot.ts) waits for the ball to come to rest or clear the deck into the pit, then the existing SettleWindow settles the rack and detectPins counts the standing pins; the drop in standing pins is the ball's pinfall, fed to Game.recordBall. The returned reset drives ResetCycle / PinSet (between-balls lift vs full re-rack), the loop advances through three balls and ten frames, and game-over shows a final-score status line with a confirm-to-restart. The live scoreboard (REQ-012) renders all ten frames, per-ball glyphs (X / spare / F flat-ten / digits), and running totals via src/scoreboard.ts into the #scoreboard overlay. The electromechanical split-flap skin (REQ-042) remains a later look-and-feel slice. Covered by tests/shot.test.ts and tests/scoreboard.test.ts.

F-004: Ball containment (gutters and a back pit)

Context
The ball slice launches the ball straight down-lane on the bed (x=0), so it stays on the lane and reaches the pins. But there are no gutters (REQ-031) on the sides and no pit/backstop behind the pin deck, so a ball with any lateral component would roll off the side and a ball that clears the rack rolls off the back of the deck and falls into the void (the ball-roll smoke confirms this). Both are scene/physics additions.
Blocker
none to start; it has no value until the ball can travel sideways.
Unblock condition
This becomes a hard prerequisite for the three-step control scheme (REQ-033, REQ-034 lateral aim and spin): gutters must exist before the ball can be aimed off-centre, and a back pit/backstop should exist so a thrown ball comes to rest rather than falling forever. Implement gutters (REQ-031) and a simple back pit before or alongside the lateral-aim controls slice.
Progress
The physical containment landed. Shared geometry helpers gutterBoxes() and pitBoxes() in src/config.ts (each box a centre plus half-extents) drive both the world3d meshes and the Rapier static colliders from one source. A recessed gutter channel (floor below the bed, inner lip, outer wall) runs each side of the bed the full lane-plus-deck length, and a recessed back pit (floor, back wall, side walls) sits behind the pin deck. A headless Rapier smoke proves a ball thrown wide drops into a gutter and is held (not lost off the side, not climbing back onto the lane), and a ball clearing the rack comes to rest in the pit on its floor rather than falling into the void. What remains for the gutter side of REQ-031: removing/scoring a gutter ball as zero pinfall belongs to the game-loop wiring (F-008), so REQ-031 stays partial.
PR / Dot reference
gutters + pit: PR #TBD
Resolved
PR #TBD. The gutter-ball scoring half landed: a new latching GutterDetector (src/gutter.ts, mirroring FoulDetector) watches the thrown ball's lateral x each watching frame; once the ball centre crosses a bed edge (abs(x) >= GUTTER.bedEdgeX = LANE.width/2) it latches guttered, and the game loop records that ball as a zero-pinfall dead ball with the rack untouched, the same dead-ball path as a foul (Q-012 default A). With the physical channels/pit already in place, REQ-031 is now fully done. Covered by tests/gutter.test.ts.

F-003: Code-split the production bundle

Context
The bootstrap production build emits a single ~2.7MB chunk (Three.js plus the inlined Rapier WASM) and trips Vite's 500kB chunk-size warning. Functionally fine for now, but worth splitting (dynamic import of the physics/render boot, manual chunks) once there is more app code to split against.
Blocker
none, but low value until more modules exist to chunk.
Unblock condition
enough app surface (gameplay systems, app shell) that splitting yields a real first-paint win.
PR / Dot reference
(not started)
Resolved
PR #TBD. The single ~2.8MB bundle is split into three cacheable chunks via Rolldown build.rolldownOptions.output.codeSplitting.groups in vite.config.ts: app entry (~45kB), three (~540kB), and rapier (~2.2MB, the inlined physics WASM). An app-code change now re-downloads only the entry chunk instead of busting the whole vendor payload. Splitting is build-time grouping, not dynamic import, so the boot path stays synchronous (main.ts pulls world3d/ball/pins eagerly) with no async-boot waterfall. The Rapier chunk is irreducibly large because the -compat build inlines its WASM and that is not splittable app code, so build.chunkSizeWarningLimit is raised to 2400kB: the warning now fires only on genuinely oversized chunks (a future regression that re-bloats a chunk past the intrinsic vendor size), not on a vendor whose size is intrinsic. Verified: production build emits the three chunks with no chunk-size or deprecation warning, the preview server boots and all three chunks resolve (HTTP 200), typecheck clean, 346 tests pass.
