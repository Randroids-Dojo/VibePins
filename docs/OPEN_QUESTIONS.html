Q-002: Physics engine, Rapier vs cannon-es

Context
The chosen stack is Three.js + Rapier (WASM). cannon-es would maximize reuse from the sibling project Hoops, but the string pinsetter needs rope/distance joints and stable stacking of ten tethered pins.
Options
  1. Rapier (WASM): modern, deterministic, strong joint/rope constraints.
  2. cannon-es: consistency with Hoops, more helper reuse, weaker constraints.
Recommended default
A.
Status
resolved
Resolution
A. Rapier (WASM), confirmed by dev on 2026-05-28 after reviewing the WASM tradeoff. The async init is a single await at boot (no per-frame cost) and Vite handles the binary; Rapier's rope joints serve the string pinsetter and its determinism supports server-side score validation. Only Hoops physics helpers are forgone; scene/render/audio/input reuse is unaffected.

Q-008: "Your turn" notification

Context
How the next player learns it is their turn when a match spans hours.
Options
  1. Link share only, plus a local "your matches" list checked by reopening the game.
  2. Web push in v1.
  3. Email or SMS in v1.
Recommended default
A.
Status
resolved
Resolution
Hybrid (A as the baseline, B as optional progressive enhancement), per dev on 2026-05-28. The always-works baseline is link sharing plus a local "your matches" list that shows whose turn it is when the player reopens the game; no permissions or providers, so v1 is never blocked on notifications. Web push is layered on as an OPTIONAL, opt-in nudge that degrades gracefully to the manual-check baseline when permission is denied or unsupported (for example iOS without an installed PWA). The web push layer itself is tracked as followup F-002 and is not a v1 blocker. Email/SMS stay out of scope.
