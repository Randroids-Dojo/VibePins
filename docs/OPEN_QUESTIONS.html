Open Questions

Questions here block or influence implementation.

Critical convention. Every question must include a <dt>Recommended default</dt> entry. The loop ships under that default and leaves the question open for override. Do not block the loop on dev sign-off. Stress-tested values that survive multiple iterations get frozen.

Q-018: Between-balls string motion, lift fallen vs recall all

Context
A playtest dot described the between-balls reset as "recall (lift) all pins, then lower only the standing ones, leaving fallen cleared". The GDD (core loop and string pinsetter) describes it differently: "only fallen pins are lifted; standing pins are left untouched (respot in place)". Both produce the identical deck end state for the next ball: the previously standing pins remain exactly where they were, and the previously fallen pins are gone. They differ only in the visible motion (lift the fallen pins, vs lift everyone then set the standing ones back down).
Options
  1. Lift only the fallen pins clear of the deck and leave them aloft; never touch the standing pins. Matches the GDD wording exactly and is the simplest consistent motion.
  2. Recall (lift) all ten, then lower only the standing pins back. Matches the playtester's mental model but moves standing pins the player deliberately left in place, against the GDD respot-in-place rule.
Recommended default
A. RULE 2: the GDD wins. RULE 7: prefer the simpler consistent flow. The end state the playtest actually requires (fallen cleared, standing untouched) is satisfied either way; A honours the GDD respot-in-place rule and does not disturb the standing cluster. Shipped under A.
Status
open
Resolution
(shipped under Recommended default A in the between-balls reset fix slice; left open for dev override)

Q-014: Concurrent-join race on a shared match seat

Context
The match backend (REQ-055) claims a seat with a read-modify-write: api/match.ts loads the match, joinMatch mutates it, then it is saved. Two players opening the same handoff link within the same instant could both read the match before either save lands and both claim the same open seat, with the second save winning. The GDD frames multiplayer as casual friends sharing one link, and a single open seat is normally claimed by one known recipient, so the window is tiny and low-stakes.
Options
  1. Accept the race for v1. Last-writer-wins on a near-simultaneous double join; in practice friends claim one seat at a time. No extra Redis round trips.
  2. Make the claim atomic with an Upstash transaction / optimistic check (WATCH or a Lua script / compare-and-set on the serialized match), retrying on conflict. Correct under contention but adds round trips and complexity for a casual flow.
Recommended default
A. The GDD scopes this to casual friends (not bank security), seats are claimed one recipient at a time, and the blast radius of a lost claim is one re-share of the link. Shipped under A. Revisit with option B if a later slice opens matches to wider/anonymous join where simultaneous claims become likely.
Status
open
Resolution
(shipped under Recommended default A in the match-backend create/join/resume slice, PR #38; left open for dev override)

Q-012: Foul scoring and rack handling

Context
Foul-line detection (REQ-032) flags an over-the-line release: the ball is at or in front of the foul line plane (z = 0, the approach side) while live. The detector exists; the question is how a foul scores. In sanctioned bowling a foul counts the ball as zero, and any pins the fouling ball knocked are reset (the rack is restored to before the throw), so the foul costs the throw but not the rack.
Options
  1. Foul scores zero pinfall for that ball and the rack is restored to its pre-ball standing state. Pins the dead ball disturbed are reeled back by the between-balls reset, so the player still has the same pins to clear on the next ball (or the next frame if it was the third ball). Matches sanctioned play.
  2. Foul scores zero but any pins the ball physically knocked stay down (no restoration), so a foul can still "help" by clearing pins for free. Simpler to implement but rewards fouling.
  3. Foul ends the frame immediately (harshest), treating the over-the-line release as forfeiting the rest of the frame.
Recommended default
A. It matches universal bowling convention (a foul is a dead ball, zero, rack restored) and stays consistent with the existing zero-count ball path in the Game spine (a foul is recorded as recordBall(0), and the returned between-balls reset reels any disturbed pins back to their home spots). Shipped under A; flip here if a target duckpin house scores fouls differently.
Status
open
Resolution
(shipped under Recommended default A in the foul-detection slice; left open for dev override)

Q-011: Tenth-frame bonus-ball rack semantics

Context
A tenth-frame strike grants two bonus balls and a spare grants one (REQ-007). The GDD says "bonus balls are thrown on a fresh re-rack as needed so the bonus pin counts are well defined." The word "as needed" is ambiguous: are the two bonus balls after a strike thrown at one shared rack (so the second hits only what the first left standing), or does the rack reset before every bonus ball (so each is an independent 0-10)? This decides whether an input like a tenth frame of [10, 7, 6] is legal (independent balls, score 23) or impossible (7 + 6 = 13 on a shared rack). It directly affects server-side rejection of impossible scores (REQ-059).
Options
  1. Standard bowling semantics: re-rack only "as needed", i.e. after a ball clears the rack. The first bonus ball is on a fresh rack; if it is a strike (10) the rack resets and the second bonus ball is again 0-10, otherwise the second bonus ball hits only the remaining pins (bonus1 + bonus2 <= 10). A tenth-frame spare's single bonus ball is on a fresh rack (0-10).
  2. Every bonus ball on its own fresh rack: each bonus ball is an independent 0-10 regardless of the previous bonus ball, so [10, 7, 6] would be legal.
Recommended default
A. This matches universal tenpin/duckpin tenth-frame convention and the most natural reading of "re-rack as needed" (the re-rack happens when a strike clears the deck, not unconditionally). The scoring engine validates under A: a tenth-frame strike's two bonus balls obey bonus1 + bonus2 <= 10 unless bonus1 is itself a strike. Shipped under A; flip to B here if a target duckpin house actually re-racks between every bonus ball.
Status
open
Resolution
(shipped under Recommended default A in the scoring engine slice; left open for dev override)

Q-017: Wire-payload validation library for the serverless backend (zod vs hand-validation)

Context
REQ-064 and the GDD async-backend spec (reuse and tech) describe the backend as serving both the leaderboard and async multiplayer with zod-validated request payloads, copying the Hoops shape. The shipped backend (api/leaderboard.ts, api/match.ts) instead validates every write by hand at the wire boundary (shape checks plus the shared pure scoreGame engine for the score line, and the pure src/match.ts transitions for match writes). zod is not a project dependency. The question: adopt zod to match the GDD text literally, or keep hand-validation and amend the requirement intent.
Options
  1. Keep hand-validation; zod stays out of package.json.
  2. Add zod and reshape both routes' payload validation around schemas to match the GDD text exactly.
Recommended default
A.
Status
resolved
Resolution
B (add zod). The earlier recommended default was A (keep hand-validation), on the grounds that adding zod brought no behavioural gain and RULE 3 reserved new deps for explicit approval. That approval was since granted: the user explicitly approved adding zod so the backend matches the GDD text ("zod-validated payloads") literally. The implementation adds api/schemas.ts and routes every request body/param in api/leaderboard.ts (POST submit, GET query) and api/match.ts (POST create/join, PATCH submitTurn) through zod schemas, returning 400 with a safe error on a malformed, oversized, or out-of-range payload. zod is the outer shape/bounds gate only; the pure scoreGame engine stays the scoring authority that ranks the board (REQ-059) and the pure src/match.ts transitions stay the turn-order/scoring authority (REQ-049, REQ-053). The existing clamps and sanitization are encoded, not loosened (sanitizeName still sanitizes the display name; zod adds a hard upper safety bound). zod is pinned at ^4.4.3 (matches the sibling Hoops) and recorded in docs/DEPENDENCY_LEDGER.html; it lives only in the serverless api/ path and is not in the Vite client bundle. Rejection paths are covered in tests/leaderboard.test.ts and tests/match.api.test.ts. REQ-064 flipped partial to done. This supersedes the prior option-A resolution recorded above.

Q-016: Which line(s) of a finished async match post to the global leaderboard

Context
When an async match completes (REQ-058), each seat has a full ten-frame line. We must decide which line(s) feed the global all-time / daily board, and which device posts them, without double-posting or posting under a guessed name.
Options
  1. Each device posts only its own seat's line, under the name it claimed the seat with.
  2. One device (e.g. the device that bowled the final frame) posts every seat's line.
  3. Only the winner's line posts.
Recommended default
A.
Status
resolved
Resolution
A. Each device posts only its own seat's line. Rationale: a device is authoritative for the name it claimed its seat with, so a line never lands under a guessed name; every seat's line posts exactly once across the participating devices (no cross-device duplicates); and a spectator with no seat posts nothing. The post is guarded by a persisted per-match flag (src/settings.ts markMatchPostedToBoard / hasPostedMatchToBoard) so re-viewing the finished match cannot re-post. The server re-scores authoritatively (REQ-059), so the client never sends a claimed total. See leaderboard, what counts.

Q-002: Physics engine, Rapier vs cannon-es

Context

Options
  1. Rapier (WASM): modern, deterministic, strong joint/rope constraints.
  2. cannon-es: consistency with Hoops, more helper reuse, weaker constraints.
Recommended default
A.
Status
resolved
Resolution
A. Rapier (WASM), confirmed by dev on 2026-05-28 after reviewing the WASM tradeoff. The async init is a single await at boot (no per-frame cost) and Vite handles the binary; Rapier's rope joints serve the string pinsetter and its determinism supports server-side score validation. Only Hoops physics helpers are forgone; scene/render/audio/input reuse is unaffected.
