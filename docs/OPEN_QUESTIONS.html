Open Questions

Questions here block or influence implementation.

Critical convention. Every question must include a <dt>Recommended default</dt> entry. The loop ships under that default and leaves the question open for override. Do not block the loop on dev sign-off. Stress-tested values that survive multiple iterations get frozen.

How to add a new question

<section data-q="Q-NNN" data-status="open">
  <h3>Q-NNN: Short title</h3>
  <dl>
    <dt>Context</dt><dd>one or two sentences on why this is a decision point.</dd>
    <dt>Options</dt><dd>
      <ol type="A">
        <li>Option A description.</li>
        <li>Option B description.</li>
        <li>Option C description.</li>
      </ol>
    </dd>
    <dt>Recommended default</dt><dd>B. One sentence on the rationale.</dd>
    <dt>Status</dt><dd>open</dd>
    <dt>Resolution</dt><dd>(filled in once dev confirms or overrides)</dd>
  </dl>
</section>

Keep Q-NNN IDs monotonically increasing. When a question resolves, leave the entry in place and update data-status to resolved plus the Resolution entry. Never delete.

Q-021: Cord-tension lift and genuine-snag shake tunables (supersedes Q-020)

Context
The reset lift was reworked to be genuinely cord-tension driven (pins reeled up by the neck, hanging and swinging under gravity, never snapped upright) and tangles were made rare and genuine (only when a downed pin lies across another pin's cord so a pin cannot reel to its clearance), per research and a product-owner playtest. This replaces the drop-and-unwind model Q-020 tuned (the releaseY / verifyFrames / pile-distance read), so those tunables no longer exist. The new feel is governed by: the lifted rope length (RESET.liftRopeLength 1.85m: how high the neck is reeled, so how the dangle reads), the genuine-snag read (TANGLE.clearanceNeckY derived from the lifted rope length, and clearanceTolerance 0.5m: how far below clearance a pin must be held to count as snagged, which sets how rarely the shake ever fires), and the up/down shake (shakeRopeLength 2.4m, shakeDownFrames 24, shakeUpFrames 24, maxRetries 4).
Options
  1. Ship the first-pass defaults above: a cord-tension lift to a 1.85m rope (the neck hangs near ~0.65m so the pin dangles clear of the deck), a snag read with a generous 0.5m clearance tolerance (so only a pin genuinely held low reads snagged and a clean rack never shakes), and a bounded up/down shake capped at 4 retries. Tighten against the playtest gate.
  2. Reel higher / lower (change liftRopeLength) to change how far the dangling pin clears the deck and how the swing reads.
  3. Tighten or loosen the snag tolerance to make genuine snags more or less frequent, trading the rarity of the charming struggle against false positives that would shake a near-clean rack.
Recommended default
A. The defaults give the signature cord-tension dangle (proven in the real-Rapier smoke: a lifting pin's centre hangs below its neck anchor and swings), a clean rack that reels straight up with no shake (the KEY REGRESSION), and a bounded recovery that always force-clears within the cap on a genuine snag. All tunable from the one RESET / TANGLE block (REQ-025); the product owner will playtest the feel. Tracked for tuning as FOLLOWUPS F-014. Supersedes Q-020 (the drop-and-unwind tunables it tuned were deleted).

Q-020: Tangle drop-and-unwind recovery tunables (drop height, settle window, retry cap, tangle threshold)

Context
The reset's tangle drop-and-unwind recovery (REQ-024) is governed by the TANGLE config block: how far the rack is lowered for the hang test (releaseY), how long it is let loose to settle before the read (verifyFrames), how many drop-and-unwind retries before a force-clear (maxRetries), and the read thresholds (pileDistance and the at-rest speeds). These set how often a recall actually struggles and how the struggle reads (how big the drop is, how long it lingers, how snappy the retries feel). Tighten against the playtest gate.
  2. Make the struggle rarer / faster (lower retry cap, shorter settle), trading charm for pace.
  3. Make the struggle more pronounced (bigger drop, longer settle, looser tangle threshold so more recalls visibly struggle), trading pace for show.
Recommended default
A. The first-pass values give a soft, legible drop that settles cleanly (a clean rack passes on the first drop with zero retries in the real-Rapier smoke) and a bounded loop that always terminates and sets the rack; the product owner will playtest the feel and the threshold can be re-tuned from the one TANGLE block (REQ-025). Tracked for tuning as FOLLOWUPS F-014.

A real string machine sets standing pins back on their home spots (the spotting fixtures define where a pin sits), and the existing reposition/lower path already carries a pin to its home spot, so A is the simplest consistent flow (RULE 7). The downed pins are still cleared, so the next ball faces the same standing pins on their home spots. Shipped under A.

Q-012: Foul scoring and rack handling

Context
Foul-line detection (REQ-032) flags an over-the-line release: the ball is at or in front of the foul line plane (z = 0, the approach side) while live. The detector exists; the question is how a foul scores. In sanctioned bowling a foul counts the ball as zero, and any pins the fouling ball knocked are reset (the rack is restored to before the throw), so the foul costs the throw but not the rack.
Options
  1. Foul scores zero pinfall for that ball and the rack is restored to its pre-ball standing state. Pins the dead ball disturbed are reeled back by the between-balls reset, so the player still has the same pins to clear on the next ball (or the next frame if it was the third ball). Matches sanctioned play.
  2. Foul scores zero but any pins the ball physically knocked stay down (no restoration), so a foul can still "help" by clearing pins for free. Simpler to implement but rewards fouling.
  3. Foul ends the frame immediately (harshest), treating the over-the-line release as forfeiting the rest of the frame.
Recommended default
A. It matches universal bowling convention (a foul is a dead ball, zero, rack restored) and stays consistent with the existing zero-count ball path in the Game spine (a foul is recorded as recordBall(0), and the returned between-balls reset reels any disturbed pins back to their home spots). Shipped under A; flip here if a target duckpin house scores fouls differently.
Status
open
Resolution
(shipped under Recommended default A in the foul-detection slice; left open for dev override)

Q-011: Tenth-frame bonus-ball rack semantics

Context
A tenth-frame strike grants two bonus balls and a spare grants one (REQ-007). The GDD says "bonus balls are thrown on a fresh re-rack as needed so the bonus pin counts are well defined." The word "as needed" is ambiguous: are the two bonus balls after a strike thrown at one shared rack (so the second hits only what the first left standing), or does the rack reset before every bonus ball (so each is an independent 0-10)? This decides whether an input like a tenth frame of [10, 7, 6] is legal (independent balls, score 23) or impossible (7 + 6 = 13 on a shared rack). It directly affects server-side rejection of impossible scores (REQ-059).
Options
  1. Standard bowling semantics: re-rack only "as needed", i.e. after a ball clears the rack. The first bonus ball is on a fresh rack; if it is a strike (10) the rack resets and the second bonus ball is again 0-10, otherwise the second bonus ball hits only the remaining pins (bonus1 + bonus2 <= 10). A tenth-frame spare's single bonus ball is on a fresh rack (0-10).
  2. Every bonus ball on its own fresh rack: each bonus ball is an independent 0-10 regardless of the previous bonus ball, so [10, 7, 6] would be legal.
Recommended default
A. This matches universal tenpin/duckpin tenth-frame convention and the most natural reading of "re-rack as needed" (the re-rack happens when a strike clears the deck, not unconditionally). The scoring engine validates under A: a tenth-frame strike's two bonus balls obey bonus1 + bonus2 <= 10 unless bonus1 is itself a strike. Shipped under A; flip to B here if a target duckpin house actually re-racks between every bonus ball.
Status
open
Resolution
(shipped under Recommended default A in the scoring engine slice; left open for dev override)

Q-018: Between-balls string motion, lift fallen vs recall all

Context
A playtest dot described the between-balls reset as "recall (lift) all pins, then lower only the standing ones, leaving fallen cleared". The GDD (core loop and string pinsetter) originally described it differently: "only fallen pins are lifted; standing pins are left untouched (respot in place)". Both produce the same deck end state for the next ball (standing pins remain, fallen pins gone); they differ in the visible motion (lift only the fallen pins, vs reel the whole rack up then set the standing ones back down).
Options
  1. Lift only the fallen pins clear of the deck and leave them aloft; never touch the standing pins. Matches the old GDD wording and is the simplest motion.
  2. Recall (lift) all ten, then lower only the standing pins back onto their home spots, leaving the downed pins aloft. Matches a real string machine and the playtester's observation at Pins Mechanical.
Recommended default
A (original). Shipped first under A in PR #48.
Status
resolved
Resolution
B (recall all, then re-set the standing pins on their home spots, leaving the downed pins aloft). The original default A was overridden: the product owner confirmed by research and by watching the machine at Pins Mechanical that real string pinsetters reel up the WHOLE rack on the between-balls respot when one or more pins were knocked down, then lower the STANDING pins back onto their HOME spots while the knocked-down pins stay reeled up and aloft, cleared out of play. The GDD wording was corrected accordingly (explicitly authorized) and the between-balls ResetCycle now reels all ten up, lowers the standing pins home, and holds the downed pins aloft (landedTargets / heldAloftTargets). The deck end state for the next ball is unchanged (same standing cluster, downed pins gone), so scoring is unaffected; only the visible motion and the re-spot of a nudged-but-standing pin changed. Where a re-spotted standing pin lands (home spot vs nudged position) is tracked separately as Q-019. Shipped in the recall-all-then-re-spot-standing slice.

Q-016: Which line(s) of a finished async match post to the global leaderboard

Context
When an async match completes (REQ-058), each seat has a full ten-frame line. We must decide which line(s) feed the global all-time / daily board, and which device posts them, without double-posting or posting under a guessed name.
Options
  1. Each device posts only its own seat's line, under the name it claimed the seat with.
  2. One device (e.g. the device that bowled the final frame) posts every seat's line.
  3. Only the winner's line posts.
Recommended default
A.
Status
resolved
Resolution
A. Each device posts only its own seat's line. Rationale: a device is authoritative for the name it claimed its seat with, so a line never lands under a guessed name; every seat's line posts exactly once across the participating devices (no cross-device duplicates); and a spectator with no seat posts nothing. The post is guarded by a persisted per-match flag (src/settings.ts markMatchPostedToBoard / hasPostedMatchToBoard) so re-viewing the finished match cannot re-post. The server re-scores authoritatively (REQ-059), so the client never sends a claimed total. See leaderboard, what counts.
