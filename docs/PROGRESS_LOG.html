Ball: Spawns and Rolls Down the Lane

PR

Changed
Added the bowling ball (REQ-028, REQ-029). New src/ball.ts with a Ball class that mirrors PinSet (owns one mesh plus one dynamic body and a sync()) and pure helpers ballSpawnPosition() and ballLaunchVelocity(). A dark polished sphere with no finger holes spawns just inside the foul line on the lane bed and is launched straight down-lane at LANE.ballLaunchSpeed with a matching forward roll, rolling into the tethered pins. Continuous collision detection is enabled so the fast ball cannot tunnel through the thin bed, deck, or pins; the ball uses GROUP.BALL collision membership, and mass, friction, restitution, and damping are new LANE tunables. src/main.ts launches the ball on boot and re-rolls it on pointerdown or Space (a minimal placeholder, not the three-step control scheme).

Tests include pure spawn/launch math and a headless Rapier suite: the ball rolls many metres down-lane while still spinning and without tunnelling, rests on the bed with its motion damped out, and (integration) a launched ball reaches the rack and topples a tethered pin. A four-lens adversarial review ran; its confirmed test-rigor findings (assert maintained roll, track minimum pin tilt over the whole sim, assert damping settles the resting ball, spawn-on-bed guard) were applied. One review finding claimed a roll-direction sign bug; re-deriving rolling-without-slip for -z motion confirmed the code (omega_x = velocity.z / r = -v/r) is correct, so that fix was rejected and the comment clarified instead.
Assumptions
The ball spawns just inside the foul line on the existing bed (the bed runs from the foul line to the head pin); an approach floor behind the foul line is a later scene slice. Launch speed 8 m/s, friction 0.25, restitution 0.1, and light damping are first-pass tunables for the playtest gate. The integration smoke caught that the design's original spawn (behind the foul line, off the bed) fell through; it was corrected to sit on the bed.

Duckpin Scoring Engine

Branch
feature/scoring-engine
PR
#TBD
Changed
Added the duckpin scoring rules as a pure, framework-free, JSON-serializable module src/scoring.ts (consumed later by the scoreboard UI, the multiplayer authority REQ-053, and leaderboard validation REQ-059). scoreGame(frames) classifies each frame as strike, spare, flat_ten, or open and computes per-frame and running cumulative scores; helpers classifyFrameMark, flattenAllBalls, and validateGameInput are exported for reuse and testing. Implements the three duckpin specifics: three balls per frame; the candlepin flat ten (cleared only on ball 3) scores a flat 10 with no bonus and is never a spare; the tenth frame grants two bonus balls for a strike, one for a spare, none otherwise. Strike and spare bonuses resolve from a flat throw tape so they span later frames and into the tenth, and running totals go null from the first pending frame for in-progress games (REQ-008). Validation rejects impossible inputs (out-of-range/non-integer pins, more than ten pins within a rack, malformed frames, wrong frame count) and never throws.
Verification
em-dash grep clean; git diff --check clean; npm run typecheck clean; npm run test 77/77 passing; npm run build succeeds (known single-chunk size warning, F-003). The 77 tests cover every frame mark, consecutive strikes, strike-then-spare, the flat-ten-is-not-a-spare distinction, all four tenth-frame outcomes, cross-boundary bonus chains, a perfect 300, an all-zero 0, all-spares 150, a realistic mixed game asserted frame-by-frame, incomplete games, eight validation-reject cases, and a JSON round-trip; expected totals were hand-verified. An adversarial four-lens review workflow (rules red-team, GDD faithfulness, test rigor, style) found no mis-scores; its confirmed test-coverage gaps (more cross-boundary cases, frame-level assertions, an empty-frame reject) were closed before this entry, and stray scratch test files left by review agents were removed.
Assumptions
Tenth-frame bonus balls re-rack only as needed (Q-011, recommended default A): a tenth strike's two bonus balls satisfy bonus1 + bonus2 <= 10 unless bonus1 is itself a strike. Shipped under that default; logged open for override.
GDD coverage
REQ-003 to REQ-008 flipped to done; REQ-001 and REQ-002 to partial (rules implemented; the playable browser frame loop is a later slice). docs/gdd/02-core-loop.html status to partial with a build log entry. Added open question Q-011.
Followups
None new. The scoreboard display (REQ-012, REQ-042) and the game-loop orchestration that calls this engine are later slices.

String Pinsetter Tether: Slack Overhead Cord Per Pin

Branch
feature/string-tether
PR
#TBD
Changed
Tethered each of the ten pins to the overhead drive point, the signature string-pinsetter mechanic. In src/pins.ts, every pin now also gets a per-pin fixed, collider-less anchor body directly above its home spot and a Rapier rope joint (a max-distance constraint, slack until taut) attaching the pin neck (a local point above the dropped belly/COM) to that anchor; a thin grey THREE.Line cord per pin tracks the neck each frame in PinSet.sync(). New TETHER config object in src/config.ts holds the tunables: anchor height 2.5m, neck offset 0.30*pinHeight, slack length 2.96m, cord colour. Slack length is derived from the worst-case anchor-to-neck distance (an outer back-row pin lying flat slid to the far deck corner, ~2.81m) plus a 0.15m margin, so the cord never restrains a natural fall (REQ-014) yet stays finite for a future reset lift (REQ-015). Exported pure helpers neckLocalAnchor() and tetherAnchorPositions().
Verification
em-dash grep clean; git diff --check clean; npm run typecheck clean; npm run test 23/23 passing; npm run build succeeds (known single-chunk size warning, F-003). Tests include a pure slack-length guard that recomputes the worst-case deck-fall distance, and a headless Rapier smoke proving (a) a struck pin's tethered fall matches its untethered fall to within 1cm with the cord never going taut, and (b) a freely dropped pin is caught at the slack length. A four-lens adversarial review workflow (correctness, GDD faithfulness, test rigor, style) confirmed the implementation; its two confirmed test-rigor findings (a missing motion lower-bound and a too-loose comparison tolerance) were fixed before this entry.
Assumptions
Anchor height 2.5m is the midpoint of the GDD 2.2-2.8m range; the neck offset 0.30*pinHeight and slack length 2.96m are first-pass tunables to tighten against the playtest gate (Q-010). REQ-014 free-fall is proven via a velocity-proxy strike; real-ball-impact validation awaits the ball slice, so REQ-014 stays partial.
GDD coverage
REQ-013 and REQ-015 flipped to done; REQ-014 to partial. docs/gdd/03-string-pinsetter.html status to partial with a build log entry.
Followups
None new. Detection (REQ-016/017) and the reset cycle (REQ-018/019) remain the next string-pinsetter slices; the anchor bodies are stored so the reset slice can reel them in.
