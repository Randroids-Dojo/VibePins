Reuse and tech

Status: partial

Overview

VibePins should not reinvent infrastructure that the sibling project Hoops (a neon arcade basketball game in this same workspace, at ../Hoops) already solved. Hoops is Three.js + cannon-es + Vite + Vercel, with a procedural Web Audio engine, swipe input, a particle system, a serverless leaderboard, and a screens/HUD/settings/tutorial/feedback scaffold. This section records what to borrow, what to adapt, and what to leave. It is design intent for reuse; the concrete sequencing lives in docs/IMPLEMENTATION_PLAN.html.

Stack and the physics-engine decision

VibePins stack: Three.js + Rapier (WASM) physics + Vite + TypeScript, deployed static to Vercel. Hoops uses cannon-es, not Rapier. The deliberate divergence is for the string pinsetter: VibePins' core system needs rope/distance joint constraints and stable, deterministic stacking of ten tethered pins (see string pinsetter), where Rapier's joints and determinism are a better fit. Confirmed (Q-002 resolved): Rapier it is. Scene-level patterns from Hoops port regardless of engine.

Scene and physics scaffolding (adapt from Hoops world3d.js)

  • A single owner module for the Three.js renderer, scene, camera, and the physics world, modeled on Hoops' World3D.
  • A central config constants object (Hoops' COURT) holding lane dimensions, pin/ball sizes and masses, camera position and look-at, and spawn points. VibePins' version (a LANE constant) is the single source of tunable geometry.
  • A collision-group bitmask (Hoops' GROUP) for ball / pin / lane / gutter / string layers.
  • Renderer setup conventions to copy: pixel-ratio cap at 2, soft shadow maps, sRGB output, scene fog for the dark venue depth.

Input plumbing (the control scheme lives in section 08)

The throw is a three-step stop-the-meter sequence (SNES Super Bowling style), specified in controls, not a swipe. What ports from Hoops is the plumbing, not the gesture: Hoops already runs an oscillating launch-power meter, so its meter timing and the Input class shape (pointer/keyboard handling, a single confirm action with callbacks) are directly reusable for stopping the spin and power meters. VibeKit's gesture helpers are largely unneeded here; the confirm action is a simple tap/click/key. Mouse, touch, and keyboard must all work (pillar 4).

Audio and particles (reuse the patterns)

  • Audio: port Hoops' procedural AudioEngine (lazy init on first gesture, resume-on-suspend, enable toggle, synthesized one-shots). Replace basketball sounds with pin clatter, ball roll, and the string-reset whir (see look and feel, audio).
  • Particles: reuse Hoops' particle system and/or the VibeKit confetti module, re-skinned as mechanical debris (sparks, steam), for the strike celebration (see look and feel, juice).

App shell (adapt from Hoops)

Hoops has a reusable meta-UI layer worth adapting: a screens/state machine (screens.js), HUD (hud.js), settings with persistence (settings.js), a first-run tutorial (tutorial.js), a feedback FAB (feedbackFab.js), a console-capture helper for debugging (consoleCapture.js), and an update banner (updateBanner.js). VibePins adapts the ones it needs; the tutorial should stay minimal per pillar 4.

VibeKit

VibeKit (@randroids-dojo/vibekit) is the workspace's shared, pre-1.0 TypeScript module library (swipe/gesture helpers, confetti, seeded rng, math, virtual-joystick, a leaderboard server entry, etc.), consumed by Hoops and siblings. VibePins depends on it the same way Hoops does (github tag-pin, e.g. github:Randroids-Dojo/VibeKit#vX.Y.Z), confirmed by Q-003. Because it is pre-1.0 and may break on any release, it goes on the dependency ledger (see docs/DEPENDENCY_LEDGER.html).

Serverless backend for multiplayer and leaderboard (v1)

Both v1 server features, async multiplayer (section 05) and the global leaderboard (section 07), reuse the Hoops backend pattern: Vercel serverless functions plus Upstash Redis, with zod-validated request payloads. Hoops already demonstrates per-mode and daily boards, name length caps, and env-var handling for both legacy KV_REST_API_* and current UPSTASH_REDIS_REST_* names; copy that shape.

  • Leaderboard: mirror Hoops' sorted-set boards under a vibepins:leaderboard namespace (all-time + daily).
  • Matches: store each async match as a JSON document under vibepins:match:<id> with a roughly one-week TTL, holding seats, per-ball results, and whose turn it is. The server is authoritative for turn order and scoring (see async multiplayer, authority).
  • Validate every write with zod and reject impossible pin counts and out-of-turn submissions.

VibeKit ships a ./server entry (kv helpers, signing, rate-limiting) that may cover some of this; evaluate it when wiring the backend.

Project conventions to mirror

  • Build/test: Vite + Vitest + tsc --noEmit typecheck, as in Hoops' package scripts.
  • A DEPENDENCY_LEDGER with a watch list (Hoops watches VibeKit); VibePins' is HTML per the spiral-html scaffold.
  • Dots-based task tracking and a features backlog, consistent with Hoops' workflow.
  • Auto-deploy to Vercel on push to main.

Build log

  • Pin rack landed. Ten duckpins spawn in the standard 1-2-3-4 triangle at the head spot with a squat cylinder collider at the belly radius and belly-heavy mass properties (low centre of mass, cylinder inertia) so pins resist toppling and transfer little energy (REQ-026, REQ-027). The three back rows recede past the head spot, so a steel pin deck (collider + mesh, top coplanar with the lane bed) was added behind the lane in world3d.ts; LANE.pinDeckDepth sizes it. Pure rack-position and mass-property helpers are exported and unit tested; a headless Rapier smoke drops the real rack onto the deck and asserts the pins rest upright.
